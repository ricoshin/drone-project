APM autopilot 관리법

기본적으로 autopilot이 끊임없이 실행되기 위하여 service형식으로 관리됨.
top명령어 실행하면 ArduCopter.elf와 apm.sh이 서비스의 일부로 존재함을 확인 할 수 있다.
기본으로 실행되는 프레임은 quad-frame(쿼트콥터)

sudo systemctl stop apm.service # 지금 바로 APM Autopilot 중지
sudo systemctl start apm.service # 지금 바로 APM Autopilot 시작
sudo systemctl disable apm.service # 부팅시 자동 실행되지 않도록 함
sudo systemctl enable apm.service # 부팅시 자동으로 실행되도록 함
sudo systemctl status apm.service

APM autopilot launch

autopilot은 /home/에 위치하는 apm.sh이라는 스크립트에 의해 launched된다. (반드시 홈에 위치해야함)
apm.sh을 편집함으로써, 어떻게 autopilot을 실행하고 어떤 vehicle의 autopilot을 실행할지 선택할 수 있다.

./<ArduCopter_executable_name.elf> -A <primary_telemetry_port> -B <GPS_port> -C <secondary_telemetry_port>

#Launch ArduCopter.elf excutable, send data through USB connection (192.168.7.1) and attach GPS to port ttyo5./ArduCopter.elf -A udp:192.168.7.1:6000 -B /dev/ttyO5 #Launch hexa frame ArduCopter.elf excutable, send data through WiFi connection (11.0.0.2) and attach GPS to port ttyo5./ArduCopter_hexa.elf -A udp:10.0.0.2:6000 -B /dev/ttyO5 #Launch ArduPlane.elf excutable, send data through MAVROS bridge (127.0.0.1) and attach GPS to port ttyo5./ArduPlane.elf -A udp:127.0.0.1:6001 -B /dev/ttyO5 #Launch Rover autopilot, use USB0 us telemetry port and WiFi as secondary telemtry port#Use GPS located at ttyo5./ArduRover.elf -A /dev/ttyUSB0 -B /dev/ttyO5 -C udp:10.0.0.2:6000

udp:10.0.0.2:6000 WiFi(Hot spot)
udp:192.168.7.1:6000 같은 공유기 밑에 있는 네트워크 컴퓨터
udp:127.0.0.1:6001 MAVROS bridge
/dev/tty05 port ttyo5 (GPS port)
/dev/ttyUSB0 USB0 (915Mhz Radio)
/dev/ttyAMA0 UART 포트

UART포트에 telemetry를 연결했을때
$APM_BIN_DIR/ArduCopter.elf -A udp:127.0.0.1:6001 -C /dev/ttyAMA0 $FLAGS 

USB포트에 telemetry를 연결했을 때

$APM_BIN_DIR/ArduCopter.elf -A udp:127.0.0.1:6001 -C /dev/ttyUSB0 $FLAGS


APM Tools

mavproxy
- MAVLink 프로토컬을 지원하는 UAV를 위한 portable한 GCS.
- 커맨드라인, 콘솔 베이스. GUI를 지원하는 플러그인도 있음.
- 컴퓨터의 수에 제한 없이 네트워크 가능.
- python, pyserial, select()함수를 지원하는 어떤 POSIX 운영체제에서도 실행가능.
- 작은 넷북에서도 동작 가능한 경량 디자인
- console, moving maps, joysticks, antenna trackers, Tab-completion(명령어 탭 자동완성) 등을 지원하는 다른 모듈들을 불러와 사용 가능
sudo pip install MAVProxy
mavproxy.py

mavextract
- 로그로부터 하나의 비행 모드를 추출함
- 비행모드를 여러번 바꿀때마다 새로운 .BIN파일을 생성
- 모드 종류 : Manual, AUTO, LOITER, FBWA, RTL, STABILIZE, LAND, STEERING,HOLD, ALT_HOLD, CIRCLE, POSITION, GUIDED, ACRO and CRUISE
- 특정 비행 모드를 추출하기 위해서는 flag를 사용 : --mode MODE
mavextract.py 1.BIN --mode LAND
Processing 2/98.BIN
Creating LAND1.bin

공중에서 보낸 총 시간을 출력.

mavflighttime.py 1.BIN

로그파일의 이해

ArduCopter로 비행 데이터를 기록하는 방법은 두가지가 있다. 방법은 다르지만 만들어낸 데이터는 매우 유사하다. (예외 존재)

Dataflash logs
BeagleBoard(업데이트 안된 듯 ㅡㅡ) 데이터 플래쉬 메모리를 사용. 비행이 끝난 후 다운로드 가능.
ArduPlane과 ArduRover의 데이터플래쉬 로그는 스타트업 직후에 생성됨. ArduCopter에서는 copter가 armed된 후에 생성.

Telemetry logs(tlogs)
APM과 컴퓨터를 telemetry link를 통해 연결했을 때는 MAVProxy에 의해 기록됨. (또는 다른 GCS에 의해)

Dataflash logs

ATT (attitude information : 자세 정보)
*단위 : centi-degree (10^(-2) degree)
- DesRoll : 조종사가 원하는 roll (왼쪽으로 기울면 음수, 오른쪽으로 기울면 양수)
- DesPitch : 조종사가 원하는 pich (앞으로 기울면 음수, 뒤로 기울면 양수)
- DesYaw : 조종사가 원하는 머리 방향 (북쪽이 0도)
- Roll : 실제 기체의 roll 
- Pitch : 실제 기체의 pitch
- Yaw : 실제 기체의 yaw

IMU (accelerometer & gyro information : 가속도계, 지자기 나침반 정보)
- GyrX, GyrY, GyrZ : 자이로 초당 회전각도 비율 (degrees/second)
- AccX, AccY, AccZ : 가속도 값 (m/s/s)

GPS
- Lat: 위도(lattitude)
- Lng: 경도(longitude)

RCOUT (RC output : RC1, RC2에 대한 pwm 출력)
- 조종기로 부터 ecs/motor/RC 출력으로 보내는 pwm 명령

CTUN (throttle & altitude information : 쓰로틀 & 고도 정보)
- Alt : 커맨드에 의한 고도(미터)? 관성 네비게이션에 의해 추정된 고도.
- BarAlt : 기압계(barometer)에 따른 고도
- DAlt : AltHold, Loiter, RTL, Auto 비행모드 일때 조종자가 의도한 고도

BARO
- Alt
- Press
- Temp : 섭씨 온도
- CRt

PM (performance monitoring)
- NLon: 10ms보다 5%이상 더 많은 시간을 소모하는 (long running) main 루프의 갯수
- NLoop: 가장 마지막 PM 메세지가 보여졌을때까지의 총 루프 갯수. 보통 1000개정도 되며, slow running loop가 15%이상 되면 절대 안됨.
- MaxT : 가장 최근까지의 루프들의 소모시간 중에 최대값. 10,000에 가까워야하지만 모터가 armed될 동안은 6,000,000까지 올라갈 수 있음.
- PMT : ground station으로부터 heart beat를 받을 때마다 증가하는 카운트
- I2CErr, INVErr, INSErr : I2C, INAV, INS 에러의 갯수. I2C에러는 I2C버스에 문제가 있음을 나타내며 이는 메인 루프의 속도를 저하시킬 수 있음.


IMU

Erle-brain에 높은 진동이 가해질 경우, APM:Copter의 가속도계에 기반한 고도와 수평 위치를 추정값이 현실과 동떨어질 수 있다.
이 것은 alt_hold 모드에서는 하늘로 치솟는 문제를 가져올 수 있고, Loiter모드에서는 방향을 잃고 표류하는 오작동을 초래할 수 있다.

진동은 로그기록에서 AccX, AccY, AccZ에서 가장 잘 관찰할 수 있다.
AccX와 AccY는 -3에서 +3 m/s/s 사이에 있고, AccZ는 -15에서 -5m/s/s 사이에 있는 것이 좋다.

가속도계 값은 콥터가 위아래로 움직일 때마다 잠깐동안 변화할 것이다. 때문에 콥터가 안정적으로 떠 있는 비행구간에 데이터를 뽑아내는 것이 좋다.
하지만 콥터가 움직이는 동안에도 잔디밭처럼 생긴 그래프의 위와 아래의 차이를 비교함으로써 진동 수준을 여전히 확인할 수 있다.
때때로 몇 가닥의 풀처럼 솟아오르는 경우는 괜찮지만 이러한 모습이 지속된다면 진짜 진동 문제일 가능성이 높다.




GPS

GPS에서 발생한 위치 오차는 APM:Copter가 스스로 잘못된 위치에 있다고 생각하게 만들어서 이를 보정하기 위해 급작스럽게 비행을 하도록 만들 수 있다.
GPS메세지의 Lat과 Lng의 그래프를 그려보면 이러한 "glitches(작은 문제[결함])"를 가시적으로 볼 수 있다. 이는 잡히는 위성 수의 감소와 HDOP의 감소로 일어날 수 있다.
mavgraph.py 30.BIN "GPS.Lat" "GPS.Lng" --flightmode=apm

만약 tlogs를 사용한다면 GPS_RAW_IT 그룹의 "eph"와 "satellites_visible" 값을 그래프화 함으로써 같은 내용을 확인할 수 있다.
1.5(150으로 표시)이나 그 이하의 hdop값이면 매우 훌륭하다. 2.0(200으로 표시)이상은 나쁜 위치 값을 나타낸다.
위성 개수가 9개 아래로 떨어지면 역시 좋지 않다. 이 두 값들이 크게 바뀌면 보통은 GPS 위치 변화를 함께 동반한다. 
Dataflash 로그의 GPS 메세지에서 "HDop"와 "NSats"를 확인하면 이를 확인할 수 있다.

*HDOP (Horizontal Dilution Of Precision - 수평 정확도 저하) : HDOP가 낮을 수록 GPS정확도가 높음. 지역, 고층건물, 고가, 다리, 날씨 등에 따라 수시로 변동. 1~7사이의 수치가 업계에서 인정하는 정상 범위임. 잡힌 위성의 수가 많다고 무조건 HDOP가 낮아지는 것은 아님.  반드시 모든 위성이 위치계산에 사용되는 것은 아닐 수 있음.


Temperature

온도는 몇 가지 센서들로부터 기록될 수 있지만, 현재로서는 BARO 데이터가 유일하게 온도 값을 포함하고 있다.
Temp 필드는 기압계(barometer)로부터 섭씨로 얻어진 온도에 대한 정보를 제공한다.
mavgraph.py 5.BIN "BARO.Temp"



Performance

Performance 모니터링이나 PM은 dataflash log에 다음과 같이 나타난다.
PM {NLon : 6, NLoop : 1000, MaxT : 23731, PMT : 9, I2CErr : 0, INSErr : 0, INAVErr : 0}

NLon : 오랜 시간을 소모하는 메인 루프의 갯수 (루프가 소모해야하는 x시간보다 5% 더 길게 소모하는 루프)
NLoop : PM 메세지가 출력되는 순간까지의 루프의 총 갯수. 보통 1000개쯤이며, 이를 통해 속도가 느린 루프의 비율을 계산 가능. (15%를 넘으면 절대 안됨)
MaxT : PM 메세지가 출력되는 순간까지의 모든 루프들의 소요시간 중에 최대값. 10,000에 가까워야 함.
            단, 모터가 armed될때는 6,000,000까지도 올라갈 수 있음.
PMT : ground station으로부터 받는 hear beat가 전달될 때마다 증가하는 카운트
I2CErr, INAVErr, INSErr : I2C, INAV, INS 에러의 갯수. I2C에러는 I2C버스에 문제가 있음을 나타내며 이는 메인 루프의 속도를 저하시킬 수 있음.

느린 루프의 갯수는 다음 명령어로 그래프화 가능 :
mavgraph.py 5.BIN "PM.NLon"



Battery Monitor in APM

CS에서 Initial Setup탭 클릭 -> Optional Setup -> Battery Monitor -> Config Tunning -> Full Parameter List에서
배터리와 관련된 파라미터 확인. 아래와 다를 경우 똑같이 수정 -> Flight Info View에서 voltage뜨는지 확인.
  • BATT_CURR_PIN 3
  • BATT_MONITOR 4
  • BATT_VOLT_PIN 5
  • BATT_AMP_PERVOLT 17.000000

*배터리 타입(mAh, voltage)이 바뀔 경우는 Battery Monitor 섹션에서 다시 캘리브레이션해줘야 함.
