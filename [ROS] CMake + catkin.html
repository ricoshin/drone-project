  1. A Quick Introduction to CMake Syntax

CMakeLists.txt파일의 문법은 주석(comment), 명령어(commands), 공백(whitespace)로 구성
주석은 #


command(arg1 arg2 arg3 ...)


공백으로 인자 구분
command는 사용자 정의(user-defined macro)이거나 내부저장 명령어(built-in)
command는 명령어의 이름, 매크로 또는 함수라고 함

include나 add_subdirectory로 추가적인 입력파일들을 사용하여 내용을 추가할 수 있음


모든 공백들(space, line feed, 탭)은 명령어의 인자를 구분할 때는 제외하고는 모두 무시한다.
인용문(쌍 따옴표, 따옴표 따위들) 사이에 있는 것은 거의 모든 언어가 그렇듯이 하나의 인자로 취급한다.
백슬래시(backslash, \)는 일반적으로 해석하는 것을 방지하기 위하여 escape character로 사용될 수 있다.
CMake는 명령어 이름의 민감하지 않아서, 쓰인 명령어 위치에 COMMAND 또는 Command로 대신하여도 무관하다.

What is CMake?
제작사인 Kitware에서는 CMake가 "open-source cross platform build system"이라고 말함. C/C++을 플랫폼에 상관없이 편하게 빌드하게 해줌.
그러나 사실 CMake가 소프트웨어를 직접 빌드하지는 않음. 사용자의 빌드 시스템(e.g. Make)에 맞게 configuration을 생성해줄 뿐임. 
대부분의 빌드 시스템에서 제공하지 않는 cross platform configuration, dependency calculation, testing, packaging, installation을 케어해줌.
한마디로 사용자의 운영체제가 사용하는 빌드 시스템에 맞게 설정을 잡아주고, 미리 설치해야하는 패키지를 체크해주어서 빌드를 쉽게하도록 도와주는 기능을 함.
(사용자의 시스템이 필요한 모든 것을 갖추었는지 확인하고 Makefile과 같은 Project file을 셋업함)

Why CMake?
CMake는 개발 환경에 맞게 Makefile이나 다양한 IDE를 위한 project를 생성해줌. 윈도우 VS / 리눅스의 GNU Make에 관계없이 빌드가능.

또한 boost와 같이 라이브러리를 찾는 툴을 가지고 있으며, 외부 라이브러리를 쉽게 include하는 기능도 있어서,
외부 dependency를 가지는 프로젝트를 빌드하는데 편리함. 경로를 하드코딩 하지 않고 find tool들을 사용하여 쉽게 끌어올 수 있음.

CMake에는 CTest라는 test driver 프로그램이 포함되어 있음. 설정만 해주면 CMake가 CTest를 위한 configuration을 알아서 만들어 줌.
CTest는 모든 테스트가 끝나면 어디서 실패하고 성공했는데 summary를 제공함. 어떤 테스트를 포함하거나 제외할건지 결정할 수 도 있음.

또한 프로젝트의 결과를 적절한 위치에 설치하도록 해주는 install target을 생성하고, 이를 CPack 유틸리티를 사용하여 패키지화 가능(tar/zip/installer)

c++ catkin library dependencies


package.xml

당신의 패키지가 catkin C++ library에 depend하고 있다면, 반드시 package.xml과 CMakeLists.txt 파일에 선언되어야 하는 dependecy들이 있을 것이다.
package.xml에 쓰는 dependency를 잘못쓰더라도 당신이 컴파일하는데는 문제없음. ROS community에 릴리즈할때 다른 사용자들이 문제가 됨.

<depend>roscpp</depend>

<build_depend> 패키지 "빌드"시에만 필요한 dependency. C++헤더나 CMake 설정 파일만 제공하는 패키지. 실행시에는 필요없음.
                           시스템 라이브러리의 경우 접미어 -dev/-devel로 끝나는 "development" 패키지(헤더, 라이브러리, configuration로 구성)나
                           모든 것이 헤더파일에 정의되어 있는 no run-time라이브러리인 경우에 주로 선언
<build_export_depend> 현재 패키지를 헤더로 export하고, 이를 다른 패키지에서 불러와서 빌드할 경우에 부여되는 의존성
                                         현재 패키지를 다른 사용자가 <build_depend>에 포함시켜 빌드를 할 경우, 여기에 선언한 패키지도 함께 포함된다.
<exec_depend>  패키지 "실행"시 필요한 dependency 선언(shared libraries, executable, python modules, launch scripts)
                            보통 rosdep으로 설치할 경우, run-time package key가 없는 경우가 많음. 대부분 development package임.
                            직접 rosdep rules에 contributing할 수도 있고, 그냥 dev package를 exec dependecy에 사용할 수도 있다.
                            
<depend> 일반적으로 이렇게 쓰면 됨. 위의 모든 선언 형태를 합친 것. 빌드타임/런타임 관계없이 모두 소화가능. 의심가면 그냥 이걸로 가면 됨.
                 단, 시스템 라이브러리의 경우 패키지의 binary installation이 "deveopment" 패키지에 종속되게 되므로 추천하지 않음. (뭔소리?)

메시지/서비스 관련 dependencies
std_msgs, sensor_msgs

Python-only(when all references are in Python) dependency들은 <exec_depend>에 적합하며, CMakeLists.txt에는 따로 명시할 필요가 없다.
sys.path에 의해 자동적으로 관리된다.





CMakeLists.txt

Catkin CMakeLists.txt파일은 일반적인 CMake command들로 대부분 구성되어 있고, 몇가지 catkin-specific 명령어들이 추가되어 있다.
  1. cmake_minimum_required()
  2. project()
  3. find_package()
  4. add_message_files(), add_service_files(), add_action_files(), all catkin-specific
  5. generate_messages(), catkin-specific
  6. catkin_package(), catkin-specific
  7. add_library(), add_executable(), target_link_libraries()
  8. install()
  9. catkin_add_gtest(), catkin_add_nosetests(), add_rostest(), add_rostest_gtest(), all catkin-specific


CMake는 package.xml에 당신이 써놓은 dependency에 대해서 전혀 알지 못한다.
제대로 컴파일을 하려면 CMakeLists.txt에 당신의 헤더와 라이브러리 레퍼런스를 정확히 선언해줘야한다.
find_package나 pkg_check_modules에 의해 찾아진(found) dependency만이 그 이하 라인에서 인식될 수 있다.
그러므로 문서의 가장 탑레벨에 기술하는 것을 추천한다.

find_package(catkin REQUIRED COMPONENTS angles roscpp std_msgs)

#include 처럼 생각하면 됨.
find_package를 부르게 되면 나중에 컴파일과 링크단계에서 사용될 CMake변수들을 정의한다. ex) ${name}_INCLUDE_DIRS and ${name}_LIBRARIES
CMake community는 변수들을 표준에 따라 정의하도록 권장하고 있지만, 몇몇 패키지들은 이를 따르지 않을 수도 있다.

CMake module을 제공하지 않는 경우도 있지만, 그 라이브러리의 development package가 제공하는 pkg-config파일을 이용하면 된다.
이를 이용하려면 PkgConfig 모듈을 먼저 불러와서 라이브러리로부터 제공되는 build flag로 접근해야 한다.
package.xml에서 <depend>나 <build_depend>태크에 정의되어 있는 패키지와 똑같이 선언하면 된다.

반드시 포함되야하는 경우(optional이 아닌 경우)에는 REQUIRED를 붙인다.

만약 CMake module이 제공되어 대부분의 C++ component들이 헤더 파일안에 완전히 구현되어 있을 경우
따로 런타임에 shared library를 필요로 하지 않는다.
그러나 그렇지 않은 경우(따로 라이브러리가 필요한 경우)에는 COMPONENTS 파라미터 뒤에 특정 모듈을 명시해줘야 한다.

pkg_check_modules(GSTREAMER REQUIRED libgstreamer-0.10)
첫번째 파라미터는 나중에 사용할 CMake 변수의 접두어를 선언하는 것이다. ex) GSTREAMER_INCLUDE_DIRS, GSTREAMER_LIBRARIES
세번째 파라미터는 라이브러리가 제공하는 pkg-config파일에서 .pc확장자를 뺀 파일이름(모듈)이며, 바로 앞에 오는 REQUIRED는 일치하는 모듈이 없으면 설정 실패를 야기한다.


include_directories(include ${catkin_INCLUDE_DIRS})
전체에서 오직 한번만 실행되어야 함. 컴파일 전에, 모든 헤더의 경로를 포함시켜줘야 함.
find_package()나 pkg_check_modules()에서 찾아준 패키지 헤더의 경로를 모두 포함시킨다.
첫번째 인수인 include는 패키지 폴더의 서브디렉토리인 include를 말하는 것임. 해당 디렉토리내의 헤더 미사용시 빼도 무방함.
모든 catkin package header dependency들은 ${catkin_INCLUDE_DIRS}로 한방에 해결됨.

catkin_package(DEPENDS Boost GSTREAMER)
전체에서 오직 한번만 콜된다. catkin-specific command. Exporting을 위한 명령어. (당신의 패키지를 import할 다른 ROS 패키지들을 위한)
모든 library build target을 export(다른 catkin package에서 불러와서 사용할 수 있도록 내보내기)한다.

Building messages, services or actions

package.xml

Your package.xml must declare a <build_depend> on message_generation, and a <build_export_depend> as well as <exec_depend> on message_runtime:

<build_depend>message_generation</build_depend>
<build_export_depend>message_runtime</build_export_depend>
<exec_depend>message_runtime</exec_depend>

Your messages services, or actions will probably include fields defined in other ROS messages, like std_msgs. Declare them like this:

<depend>std_msgs</depend>

The <depend> tag is recommended for message dependencies. That example assumes std_msgs is the only dependency. Be sure to mention all your message package dependencies here, and substitute them all for std_msgsin the examples that follow.

To generate actions, add actionlib_msgs as a dependency:

<depend>actionlib_msgs</depend>

CMakeLists.txt

For CMake, find the catkin packages for message_generation and any messages, services or actions you depend on:

find_package(catkin REQUIRED COMPONENTS message_generation std_msgs)

For building actions, include actionlib_msgs among the dependencies:

find_package(catkin REQUIRED
             COMPONENTS
             actionlib_msgs
             message_generation
             std_msgs)

Next, list your message definitions:

add_message_files(DIRECTORY msg
                  FILES
                  YourFirstMessage.msg
                  YourSecondMessage.msg
                  YourThirdMessage.msg)

Similarly, if you have a service to generate:

add_service_files(DIRECTORY srv
                  FILES
                  YourService.srv)

To generate actions, add:

add_action_files(DIRECTORY action
                 FILES
                 YourStartAction.action
                 YourStopAction.action)




add_executable(your_node src1.cpp src2.cpp src_etc.cpp)
실행파일 생성을 위한 명령어. 헤더를 제외하고 모든 필요한 C++ 소스파일들을 나열.

set(${PROJECT_NAME}_SOURCES
    src/file1.cpp
    src/file2.cpp
    src/file3.cpp
    src/file4.cpp
    src/file5.cpp
    src/file6.cpp
    src/file_etc.cpp)
add_executable(your_node ${${PROJECT_NAME}_SOURCES})
소스파일 리스트가 긴 경우에는 set이용하여 변수선언.

target_link_libraries(your_node
                      ${catkin_LIBRARIES}
                      ${Boost_LIBRARIES}
                      ${GSTREAMER_LIBRARIES})
만약 빌드할 프로그램이 다른 catkin package들이 제공하는 라이브러리나 시스템 라이브러리에 종속되는 경우, target_link_libraries()에 포함시킨다.

install(TARGETS your_node
        RUNTIME DESTINATION ${CATKIN_PACKAGE_BIN_DESTINATION})
ROS 실행파일은 운영체제의 글로벌 bin/ 디렉토리가 아니라 해당 패키지 디렉토리에 인스톨된다. 이로써 shell의 $PATH에 난잡하게 채워넣지 않고도
rosrun이나 roslaunch에서 접근 가능하다. 단, 각각의 패키지는 고유한 이름을 가져야한다.
글로벌 bin/ 디렉토리에 설치되는 ROS 명령어는 rosrun이나 roslaunch 같은 일부 명령어 뿐이다.
TARGETS로써 실행파일 리스트를 명시해준다.

install(TARGETS your_library your_other_library
        ARCHIVE DESTINATION ${CATKIN_PACKAGE_LIB_DESTINATION}
        LIBRARY DESTINATION ${CATKIN_PACKAGE_LIB_DESTINATION}
        RUNTIME DESTINATION ${CATKIN_GLOBAL_BIN_DESTINATION})

ROS 라이브러리는 ROS 배포판 전체에서 공유하는 lib/ 디렉토리에 인스톨된다. 중복 네이밍을 주의한다.
중복을 피하기 위해서는 라이브러리 타켓 네임에 패키지 네임의 일부를 포함시키는 것이 좋다. (패키지 네임이 이미 고유성을 가지고 있으므로?) 
윈도우에서 .dll의 경우는 runtime destination이 글로벌 bin 폴더여야한다.
라이브러리는 보통 그들의 인터페이스를 정의하는 헤더파일을 제공한다. 표준에 따라 모든 외부 헤더파일을 include/{package_name}에 위치시킨다.

install(DIRECTORY include/${PROJECT_NAME}/
        DESTINATION ${CATKIN_PACKAGE_INCLUDE_DESTINATION})

이 커맨드는 include 서브트리에 패키지의 모든 파일을 설치하도록 한다.  그곳에 오직 exported header만 위치시켜라.
메시지, 서비스, 액션 타겟의 경우는 catkin이 자동으로 인스톨하므로 별도의 install()명령어는 필요하지 않다.

catkin에서 라이브러리는 전체 ROS distribution에서 모든 패키지가 공유하는 디렉토리에 인스톨된다.
그러므로 라이브러리 이름은 서로 충돌이 일어나지 않도록 고유하게 정해줘야 한다.


your_package가 your_library라는 shared library build target을 가진다고 가정할때
실제 파일 이름은 libyour_library.so가 된다.

add_dependencies(your_program ${catkin_EXPORTED_TARGETS})
add_dependencies(your_library ${catkin_EXPORTED_TARGETS})

타겟이 빌드되기전에 먼저 빌드되어야하는 헤더들을 지정.

서로 연관이 있는 메세지, 서비스, 액션은 다른 API없이 독립적인 패키지로 묶는 것이 좋다. 패키지 dependency graph가 단순화되도록 하는 것이 좋다.
그러나 스크립트와 프로그램을 메세지 패키지와 함께 제공할 수도 있다.
만약 그렇게 한다면, message generation target들은 그들을 의존하는 다른 어떤 프로그램보다 앞서서 빌드되어야 한다.

add_dependencies(your_program ${catkin_EXPORTED_TARGETS})

Variable names

For custom variables, avoid the following list of suffixes in any CMakeLists.txt, those should only be set by FindXXX.cmake or XXXConfig.cmake files:

  • XXX_DEFINITIONS
  • XXX_EXECUTABLE
  • XXX_INCLUDE_DIRS
  • XXX_INCLUDE_DIR
  • XXX_YY_INCLUDE_DIR
  • XXX_LIBRARIES
  • XXX_LIBRARY_DIRS
  • XXX_LIBRARY
  • XXX_YY_LIBRARY
  • XXX_ROOT_DIR
  • XXX_FOUND
  • XXX_YY_FOUND
  • XXX_RUNTIME_LIBRARY_DIRS
  • XXX_VERSION_STRING
  • XXX_VERSION_MAJOR
  • XXX_VERSION_MINOR
  • XXX_VERSION_PATCH
  • XXX_VERSION_YY
  • XXX_WRAP_YY
You may use such variables of course by reading their value after calling find_package(), but do not manually change them.
