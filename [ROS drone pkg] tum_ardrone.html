[ROS drone pkg] tum_ardrone

tum_ardrone
http://wiki.ros.org/tum_ardrone

이 패키지는 다음 논문 내용을 구현한 패키지임.

  • Scale-Aware Navigation of a Low-Cost Quadrocopter with a Monocular Camera (J. Engel, J. Sturm, D. Cremers)

  • Camera-Based Navigation of a Low-Cost Quadrocopter (J. Engel, J. Sturm, D. Cremers)

  • Accurate Figure Flying with a Quadrocopter Using Onboard Visual and Inertial Sensing (J. Engel, J. Sturm, D. Cremers)

개요

- ground-based 랩탑 컴퓨터와 연결된 저가형 쿼드콥터가 처음 가보는 미지의 지역에서 GPS 없이 자동으로 비행하는 시스템
- 세 가지 요소로 이루어져 있음 : monocular SLAM 시스템, 데이터 융합과 상태 예측을 위한 확장 칼만필터, 스티어링 명령을 생성하기 위한 PID 컨트롤러.
- 관성과 고도 측정에 의해 생성된 비주얼 맵의 절대 스케일을 예측하기 위한 새로운 closed-form 솔루션
- 인공적인 마커나 외부 센서가 없이도 절대 스케일의 미지의 환경을 탐사할 수 있음.
- 비주얼 트랙킹을 일시적으로 놓치거나 통신에 심각한 지연이 생기더라도 어느정도 복구 가능
- 비주얼 SLAM을 사용하여 오도메트리의 드리프트 현상을 제거
- 스케일을 인지하는 정확한 자세 예측과 네비게이션

코드는 AR.Drone 1.0과 2.0 모두를 위해 동작하지만, 현재 기본 파라미터는 AR.Drone2.0을 위해 최적화 되어 있다. 디테일 확인 : website.
매우 잘 알려진 모노큘러 슬램 프레임워크인 PTAM 기반으로 빌드되었음. (paper / website / license 비상업용)

노드

  1. drone_stateestimation
          : 상태 예측(state-estimation). PTAM + Visualization
  1. drone_autopilot
          : 드론 컨트롤러. drone_stateestimation이 먼저 실행되어 있어야 함.
  1. drone_gui
          : 조이스틱이나 키보드로 드론 조종하는 GUI. stateestimation과 autopilot 노드를 위한 GUI

설치

본 패키지를 사용하기 위해서는 ardrone 드라이버 패키지인  ardrone_autonomy을 먼저 설치하도록 되어있음.
현재 약간의 수정을 하여 사용해야하는데,  레포지토리 (repository)를 통해서 이미 수정해서 검증된 버전을 다운받을 수 있음. 다만 약간 오래된 버전임.
우리 프로젝트에서는 본 패키지와 ardrone 드라이버 사이의 연결고리를 찾아내어 Erle-copter와 연동될 수 있도록 수정해야함.

실행순서
 
# 1. ardrone_autonomy 드라이버 실행
rosrun ardrone_autonomy ardrone_driver

# 2. stateestimation 노드 실행
rosrun tum_ardrone drone_stateestimation

# 3. autopilot 노드 실행
rosrun tum_ardrone drone_autopilot

# 4. gui 노드 실행
rosrun tum_ardrone drone_gui

사용법

<수동 키보드 제어>

- drone-gui 창에 포커스를 놓고 ESC 누르면 키보드 제어 활성화
- 자동비행 오작동시 ESC를 누르면 바로 오토파일럿 해제하고 수동 조작 가능
- 조종법 :
     q/a : 상승/하강
     i/j/k/l : 수평방향 비행
     u/o : yaw 회전
     F1 : 비상상황 토글
     s : 이륙
     d : 착륙

<수동 조이스틱 제어>

PS3 6축 컨트롤러일 경우 가정:
- rosrun joy joy_node
- PS 버튼 눌러서 조이스틱 활성화
- 조종법 :
     왼쪽 조이스틱은 수평 컨트롤 / 오른쪽 조이스틱은 높이와 yaw컨트롤 (RC 조종기와 좌우 반대)
     L1 누르면 이륙 / L1을 놓으면 착륙
     R1은 비상상태 토글
- 양쪽 조이스틱 중 아무거나 움직이면 Control Source가 바로 조이스틱으로 넘어옴. (autopilot 오류시 안전 위해서)

<오토파일럿>

- 드론을 열린 공간에 지상 배치. 카메라로부터 2미터에서 10미터 거리 앞에 충분한 키포인트를 가진 구조물이 있어야 함(권장)
- 테스트 비행 : iniDemo.txt 파일의 내용 불러오기 -> Reset & Clear & Send 버튼 클릭
- 항상 ESC키나 조이스틱에 손을 올려놓고 긴급상황에 컨트롤을 다시 가져올 준비를 할 것.


문제 해결

불안정한 autopilot의 경우
- 제어 파라미터 조정
     - agressiveness 감소 : 더 느리지만 안정적으로 비행
     - 롤/피치 제어의 differential weights(Kp_rp) 감소 
- 키포인트가 너무 멀리 존재하는 경우에는 PTAM 포즈 예측이 부정확할 수 있음

PTAM 초기화 실패 에러 메세지
- PTAM을 초기화하기 위해서는 stateestimation 노드에서는 미터 스케일 정보를 잡아주어야 한다.
  (수직 속도 예측이나 초음파 고도측정)
- 이러한 정보는 비행중일때만 받을 수 있으므로, PTAM 초기화는 비행 중에만 실행될 수 있다.
- 실제로 비행하지 않고도 record한 것을 리플레이 하는 방식으로 테스트해볼 수 있다.

Navinfo에 관한 에러 메세지
- 보통 navdata_debug는 0으로 설정되어 있으며, 이때 새로운 navdata 패키지는 5초마다 전송된다.
- 에러 메세지는 일정 시간동안 navdata를 전송받지 못함을 알리는 것이며, 이러한 현상이 계속되면 상태 예측과 비행 성능이 저하됨을 의미한다.
- /ardrone/navdata의 발행 주파수 (UI윈도우에 표시됨)가 50근처라면, 드라이버의 loop_rate가 잘못 설정된 것이고,
   만약 15 정도라면, navdata_demo가 1로 설정되어 있는 것이다.
- 다른 이유는 네트워크 연결 불량 또는 높은 ping 등이 있을 수 있으며, 드론을 재시작하거나 네트워크 환경이 좋은 곳으로 이동해야 한다.

드론이 캐박살나는 경우
- 새로 산다 -_-

주의

/tum_ardrone/com은 노드간의 통신을 위해 사용되는 채널이다. 내부적으로 무슨 일이 일어나는지 알고 싶으면 echo 명령을 이용하라.

알려진 버그와 이슈

- 이 소프트웨어는 원래 기압 센서와 자력계가 없는 Ar.Drone 1.0을 위해서 개발되었던 것이다. --> 초음파 고도계 사용한 이유인 듯. 우리는 둘 다 사용 가능.
- PTAM 코드 자체에서 초기화가 실패하는 경우 drone_stateestimation에서 에러발생가능.
  특히 처음 두개의 키프레임 사이에 베이스라인이 존재하지 않을 경우 발생가능.


tum_ardrone/drone_stateestimation

개요

받은 navdata와 제어 명령, 그리고 PTAM에 기반하여 드론의 위치를 예측한다.
다음 두 채널로부터 메세지를 받아야 한다.
  1. /adrone/navdata (100Hz이상)
  2. /ardrone/image_raw (10Hz이상)
직접 연결되어 ardrone_autonomy노드를 실행하고 있는 드론이나 .bag 리플레이로부터 받을 수 있음.

ardrone_automy 실행 옵션
rosrun ardrone_autonomy ardrone_driver _navdata_demo:=0 _loop_rate:=500
PTAM 스케일을 잘 예측하려면 초기화 직후에 드론을 위아래로 약간씩(1m정도 업다운) 움직여주는 것이 필요하다. 
이러한 테스크를 위해서 아래 논문의 방법이 사용된다.

  • Camera-Based Navigation of a Low-Cost Quadrocopter (J. Engel, J. Sturm, D. Cremers)

  • Accurate Figure Flying with a Quadrocopter Using Onboard Visual and Inertial Sensing (J. Engel, J. Sturm, D. Cremers)
이 노드는 다음을 구현한다.
- 상태-예측을 위한 EKF (드론의 동역학 모델 포함)
- 시간 지연에 대한 보상
- 스케일 예측
- PTAM 통합

기능

두개의 창으로 구성됨. 하나는 카메라 영상과 PTAM의 맵 포인트를 보여주고 다른 하나는 지도를 보여준다.
주요 명령어를 수행하도록 하려면 각각의 윈도우를 활성화 시킨뒤 해당 키를 누른다.
이는 /tum_ardrone/com에 명령어를 저장하도록 하고, 이는 차례로 파싱되어 해당 작업을 수행한다.

<비디오 창>
R : PTAM 리셋
U : 디버그 정보 디스플레이(토글)
Space : 초기화를 위하여 첫번째와 두번째 키프레임을 촬영
K : 강제로 키프레임을 촬영하도록 함
L : 로그 생성 토글
M : 지도를 잠금. PTAMMapLock 파라미터와 같음. dyn. 설정 파라미터에 의해 덮어쓰기 가능.
N : 씽크를 잠금. PTAMSyncLock 파라미터와 같음. dyn. 설정 파라미터에 의해 덮어쓰기 가능.

마우스로 타겟 세팅
- 비디오 윈도우에서 마우스 클릭으로 웨이포인트를 생성하여 drone_autopilot으로 전송한다.
- 좌클릭 : 현재 위치에서부터 (x,y,0)미터 이동. 이미지 중앙은 (0.0)이고 경계선은 각각 2미터이다.
- 우클릭 : 현재 위치에서 (0,0,y)미터 이동해서 x도 만큼 yaw를 회전. 이미지 중앙은 (0,0)이고 경계선은 각각 2미터에 90도의 각도를 갖는다.

<지도 창>
R : EKF와 PTAM 리셋
U : 디버그 정보 디스플레이 토글
V : 뷰어의 시점을 리셋
L : 로그 파일 기록 시작/정지
V : 드론 이동 경로 기록 삭제 (녹색 선)

PTAM 성능 보장을 위한 팁

- 많은 구조물이 있는 실내에서 가장 잘 작동함
- 나무나 식물에는 잘 작동하지 않음. (특히 바람이 부는 경우). PTAM은 움직이지 않는 정적 세계(static-world)를 가정하는데 이에 위배됨.
- PTAM은 적절한 수직이동이 없이 회전만 하면 트랙킹이 잘 이루어지지 않음. 드론의 yaw를 너무 많이 변화시키지 않도록 노력해야함.
- 키포인트는 2미터에서 10미터에 위치하는 것이 이상적임. 더 가까운거리에서는 드론이 너무 빨리 움직이고, 더 먼거리에서는 포즈 예측이 부정확해짐.
- 이상적으로는 키포인트가 여러 뎁스에서 발견되어야 함. 하나의 벽면과 같은 동일한 평면위에 존재하는 키포인트는 시야각이 좁을 때에는 성능에 좋지 않음.
   (수직적 움직임과 피치회전을 구별하는 것이 매우 어려워지기 때문임)로
- 위아래로 움직여주는 것은 스케일 예측을 더 정확하게 해줌. 미터 정보를 얻어내는 주요 정보원이 초음파 고도계이기 때문임. 
  수평으로만 움직이게 되면(특히 평탄하지 않은 지형 위에서 -고도가 무의미해짐-) 스케일 예측이 잘 되지 않음. 당연히 비행 성능도 저하됨.
  --> 우리 케이스의 경우 기압계를 대체하여 사용할 수 있는지 확인!

비행 기록과 재생

수동 비행으로부터 3가지 데이터 기록
rosbag record -O flight.bag /ardrone/image_raw /ardrone/navdata /cmd_vel
기록한 데이터를 재생
rosbag play -l flight.bag

이를 통해 드론이 박살날 염려따위 없이 편안하게 stateestimation을 테스트해볼 수 있다!
ardrone_driver노드를 끄고, Control Source를 "None"으로 설정(또는 drone_gui 종료)하는 것을 잊지말아라.
그렇지 않으면 데이터를 재생할때, 3개의 데이터들이 실제 데이터들과 섞여서 나타나게 된다.

파라미터

~publishFreq: 드론의 예측된 위치가 계산되어서 퍼블리쉬되는 빈도(주파수). 기본값 30Hz.
~calibFile : 카메라 보정 파일. 세팅이 안되있으면 기본값이 사용됨. (camcalib/ardroneX_default.txt)


토픽

reads
  • /ardrone/navdata
  • /ardrone/image_raw
  • /cmd_vel
  • /tum_ardrone/com -> 읽기/쓰기 모두

writes
  • /ardrone/predictedPose
  • /tum_ardrone/com -> 읽기/쓰기 모두

동적으로 재설정가능한 파라미터들

dynamic_reconfigure 패키지를 사용하여 노드의 재시작없이 동적으로 파라미터를 수정할 수 있도록 함.

UseControlGains : EKF 예측을 위해 제어 게인을 사용할지 여부
UsePTAM : EKF가 업데이트될 때마다 PTAM 포즈 예측을 사용할지 여부
UseNavdata : EKF 업데이트를 위해 Navdata를 사용할지 여부
*만약 UsePTAM과 UseNavdata가 false로 세팅되어 있다면, EKF는 절대로 업데이트 되지 않고, 완전한 시뮬레이터처럼 작동한다.
 컨트롤 명령(/cmd_vel)을 기반으로 해서 예측한 결과로만 시뮬레이션하므로 실험을 하기에는 좋다.

PTAMMapLcok : PTAM 지도를 잠금 (KF를 더이상 사용하지 않음)
PTAMSyncLock : PTAM 지도 싱크를 잠금 (스케일과 포즈 오프셋 등을 고정시킴)
PTAMMaxKF : PTAM에 사용되는 최대 KF량.
PTAMMinKFDist : 두 개의 KF사이의 최소 거리 (미터)
PTAMMinKFWiggleDist : 두 개의 KF사이의 최소 거리 (평균 씬 뎁스에 비교한 상대값)
PTAMMinKFTimeDiff : 두 개의 KF 사이의 최소 시간
*다음의 조건을 만족할 때 새로운 KF를 취함 : PTAMMinKFTimeDiff && (PTAMMinKFDist || PTAMMinKFWiggleDist) --> 트랙킹이 좋을 때의 조건?
*KF --> Key Frame을 말하는 듯. 논문을 읽어보면 나옴.. 일단은 pass..

RescaleFixOrigin : 지도의 스케일이 다시 예측되는 경우, PTAM과 실제 물리세계를 매핑한 점들 중 오로지 한 점만이 고정되고 나머지는 변화함. - false일 때는 드론의 현재 포즈를인 기준으로 재예측. 드론의 갑작스런 점프를 막아주는 대신에 지도가 "드리프트"된다.
- true일 때는 두번째 KF가 얻어진 초기 위치를 기준으로 재예측. 드론의 위치는 갑자기 점프할 수 있지만 지도는 고정된다.

c1 ... c8 : EKF의 예측모델 파라미터. 논문 참조.

tum_ardrone/drone_autopilot

일반

드론 제어를 위한 PID 컨트롤러. 기본적인 웨이포인트 추종과 자동 초기화를 포함한다.
drone_stateestimation이 먼저 구동하고 있어야만 한다. 목표는 /tum_ardrone/com 토픽을 통해서 설정된다.

본 노드는 다음을 구현한다.
  • Parrot AR.Drone을 위한 PID 제어
  • 자동 초기화 과정 / autopilot을 제어하기 위한 언어 스크립팅

언어 스크립팅 : 

autopilot의 구동시키려면 /tum_ardrone/com에 명령어를 보내야한다. 그 형식은 "c COMMAND"이다.
명령어는 대기열(큐)에 따라 차례로 실행되며, 하나의 명령어가 끝나자마자 다음 명령어가 팝(pop)된다.
예를 들어, 하나의 웨이포인트에 도달함과 동시에 다음 웨이포인트로 가는 명령이 수행되는 식이다.
대기열을 비우려면 "c clearCommands"를 전송하면 된다. 명령은 drone_gui 노드를 사용해서 전송할 수 있다.
명령어는 /flightPlans/*.txt에 적어서 한번에 대기열로 보낼 수 있으며, 사용할 수 있는 스크립트 명령 예시는 다음과 같다.

autoInit [int moveTimeMS] [int waitTimeMS] [int riseTimeMs] [float initSpeed]
        takes control, starts drone, initializes PTAM. That is:
          제어권을 가져오고, 드론을 시동하고, PTAM을 초기화 한다. 즉, :
          - 드론 시동하고 riseTimeMs(밀리초)동안 기다린 뒤에 1미터 정도 띄운다.
          - 첫번째 KF를 획득하고, initSpeed의 속도로 moveTimeMS동안 날아오른 뒤에, waitTimeMS만큼 다시 기다린 뒤, 두번째 KF를 획득.
          - 성공하지 못하면, 오르락 내리락 하면서 계속해서 시도함.
          - 괜찮은 초기 값은 "autoInit 500 800 4000 0.5" 

autoTakeover [int moveTimeMS] [int waitTimeMS] [int riseTimeMs] [float initSpeed]
        위의 autoInit과 똑같다. 단, 이미 비행하고 있을 때 사용된다. (첫 번째 이륙 단계를 제외)

takeoff 
          - 제어권을 가져오고, 드론을 시동하는 것까지만 수행.
          - 맵을 리셋하거나 PTAM을 초기화하지는 않음.

setReference [doube x] [double y] [double z] [double yaw]
        기준좌표 설정 (이 점을 기준으로 하여 goto X X X X 명령어로 이동).
 
setMaxControl [double cap = 1.0]
          최대 제어 전송량이 [cap]으로 제한됨. 드론을 더 느리게 날도록 함.
       
setInitialReachDist [double dist = 0.2]
          드론이 웨이포인트에 도달한 것으로 인정되는 거리 범위

setStayWithinDist [double dist = 0.5]
          드론이 웨이포인트에서 일정시간(setStayTime) 동안 머무르면서 유지해야하는 거리
        drone has to stay this close to a way point for a certain amount of time.
       
setStayTime [double seconds = 2.0]
         드론이 웨이포인트에서 일정거리(setStayWithDist) 안에서 머물러야하는 시간
        
clearCommands
          모든 목표지점을 지워서 다음 명령이 즉시 실행되도록 한다.

goto [doube x] [double y] [double z] [double yaw]
          현재 기준점(setReference 설정값)에 대한 특정 위치(x,y,z yaw)로 드론을 보낸다.
                  위의  setXXX 파라미터에서 설정한대로 목표를 달성할때까지 블로킹상태가 된다.

moveBy [doube x] [double y] [double z] [double yaw]
          현재 목표지점에 대한 특정 위치(x,y,z yaw)로 드론을 보낸다.
                  위의  setXXX 파라미터에서 설정한대로 목표를 달성할때까지 블로킹상태가 된다.

moveByRel [doube x] [double y] [double z] [double yaw]
          현재 예측되는 드론의 위치에 대한 특정 위치(x,y,z yaw)로 드론을 보낸다.
                  위의  setXXX 파라미터에서 설정한 대로 목표를 달성할 때까지 블로킹상태가 된다.
       
land
          자동 착륙. 초기화.
       
lockScaleFP
          스케일이 재예측될 때 변화하지 않는 하나의 지점을 현재 드론의 위치로 지정.
          PTAM이 잘되고 있을때만 설정가능하다. (아니라면, PTAM이 초기화되고 잘 동작할 때까지 지연된다.)
          동적 재설정 파라미터인 useWorldFixPoint를 세팅해주어야만 사용 가능.

파라미터

~minPublishFreq : 보통 제어 명령은 drone_stateestimation으로부터 받은 각각의 포즈 예측에 대하여 보내진다. 
그러나, minPublishFreq(밀리초)이상의 시간동안 포즈 예측이 전달되지 않으면, HOVER 명령이 보내진다. 기본값 : 110.
(갑자기 포즈 예측이 작동하지 않더라도 드론이 호버링을 유지할 수 있도록함.)

동적 재설정 파라미터들

- Ki_X, Kd_X, Kp_X : 롤/피치, gaz?(위/아래), 요(yaw)를 위한 PID 제어 파라미터
- max_X : maximal respective control command sent (ever).
- rise_fac : 이 인수에 의해서 상승 명령이 하강 명령보다 더 커지도록 해준다. (기본적으로 떨어지는 것은 매우 빠르지만 올라가는 것은 큰 힘이 필요하기 때문
- aggressiveness : 보내지는 모든 명령어의 PI 요소에 곱해준다. 값이 낮으면 드론이 천천히 날게 된다.

사용되고 있는 토픽 모니터링

우상단에는 현재 중요한 토픽의 현재 발행 빈도(Hz)를 보여준다.
- Drone Navdata : /ardrone/navdata를 감시. 드론 연결하고 ardrone_autonomy 노드 실행했을 때, 150~200Hz 정도 나와야함.
- Drone Control : /cmd_vel을 감시. 이는 얼마나 자주 컨트롤 명령이 발행publish 되는지(드론쪽에서 구독subscribe)에 관한 빈도이다.
- Pose Estimates : /ardrone/predictedPose를 감시. drone_stateestimation 노드에 의해 생성된 상태 예측이다. 기본값 30Hz이다.
- Joy Input : /joy를 감시. 게임패드를 연결하여 joy_node를 실행하고 있다면 0Hz이상의 값이 뜬다.
- Ping(RTT) : 500B와 20kB 패키지에 대한 현재 무선 LAN 지연도(latency - RTT in ms). 너무 높으면 Wlan clutter를 줄여야함.

Autopilot과 Stateestimation 노드의 모니터링

오른쪽 중간부에는 autopilot과 stateestimation 노드의 현재 상태를 보여줌 (1초에 한번씩 새로고침됨)

현재 컨트롤 소스 선택

Control Source 항목에서 어떻게 드론을 컨트롤할 것인지를 선택할 수 있음. Autopilot선택시, autopilot node는 오직 제어 명령만을 보낼 것이다.

오토파일럿 제어

drone-gui창에서 왼쪽의 텍스트 필드를 사용하여 autopilot node에 명령 전달. (예를 들어, 웨이포인트 궤적)
  1. 좌상단의 텍스트 필드에 명령어 쓰기 (또는 flightplans/ 디렉토리 밑에서 txt파일 형태로 불러오기 가능)
  2. Send 버튼 : autopilot의 명령 큐로 커맨드 전송 (자동으로 Control Source가 Autopilot으로 세팅됨)
  3. Clear버튼 : autopilot 명령어 큐를 와 현재 목표를 지움
  4. Reset버튼 : Autopilot, PTAM, EKF를 재설정
